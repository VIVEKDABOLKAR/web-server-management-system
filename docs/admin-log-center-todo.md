## Phase 3 - Backend Log APIs (By Type and Date)
Goal: Serve logs safely to admin UI.

### API Endpoints (suggested)
- [ ] `GET /admin/logs/types` - list available log types.
- [ ] `GET /admin/logs?type={type}&date={yyyy-mm-dd}&tail=10` - fetch last N entries.
- [ ] `GET /admin/logs/download?type={type}&date={yyyy-mm-dd}` - download full file.

### API Behavior
- [ ] Validate `type` against allow-list.
- [ ] Validate date format and supported date range.
- [ ] Return stable error model for missing file or invalid request.
- [ ] Add pagination or incremental read strategy for large files.
- [ ] Add guardrails: max `tail`, max bytes returned.

### Security and Performance
- [ ] Admin authorization on all log endpoints.
- [ ] Prevent path traversal and arbitrary file access.
- [ ] Add rate limit/throttle for download endpoint.

### Exit Criteria
- [ ] Type/date/tail API returns correct last entries.
- [ ] Download endpoint returns file with correct headers.
- [ ] Security checks pass (no arbitrary path access).

## Dependencies and Order (Critical Path)
- [ ] Phase 1 complete before Phase 3 (WebSocket production readiness first).
- [ ] Phase 2 complete before Phase 3 (log files must exist before serving).
- [ ] Phase 3 complete before Phase 4 (frontend depends on APIs).
- [ ] Phase 4 complete before Phase 6 (release).

## Risks and Mitigations
- [ ] Risk: large log files cause slow response -> Mitigation: tail/max-bytes limits.
- [ ] Risk: path traversal vulnerability -> Mitigation: strict allow-list + canonical path checks.
- [ ] Risk: secrets leaked in logs -> Mitigation: masking and logger policy review.
- [ ] Risk: production WS mismatch (`ws` vs `wss`) -> Mitigation: env validation + startup checks.
